# 12 — 架构决策记录（Architecture Decision Records）

> 文档导航：返回 [docs/README.md](README.md)。

---

## 1. 什么是 ADR

**架构决策记录 (ADR)** 是一种轻量级文档格式，用于记录在软件开发过程中做出的重要技术决策。每条 ADR 回答三个问题：

1. **背景**：为什么需要做这个决策？
2. **方案**：考虑了哪些选项？
3. **决策**：最终选了什么？为什么？

ADR 是**不可变的**——一旦决策被记录，不应该修改原文。如果决策被推翻，应新增一条 ADR 标记原决策为"已废弃"。

---

## 2. ADR 模板

新增 ADR 时使用以下模板：

```markdown
### ADR-{NNN}: {标题}

- **日期**：YYYY-MM-DD
- **状态**：✅ 已采纳 | ❌ 已废弃 | 🔄 被 ADR-XXX 取代
- **决策者**：{谁做的决策}

**背景**
{为什么需要做这个决策？存在什么问题或需求？}

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| 方案 A | ... | ... |
| 方案 B | ... | ... |

**决策**
{最终选择了哪个方案？为什么？}

**影响**
{该决策对后续开发有什么影响？}
```

## 3. 决策记录

### ADR-001: 采用 Maven 多模块聚合架构

- **日期**：2026-02-23
- **状态**：✅ 已采纳
- **决策者**：架构团队

**背景**
V1/V2 时期，App 和 Admin 后端各自独立 POM，导致大量代码重复（Entity、Service、Security、Redis 等）。维护成本高，Bug 修复需要改两处。

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| A: 保持独立 POM | 部署独立、互不影响 | 代码重复严重、Bug 需双修 |
| B: Maven 多模块聚合 | 共享代码、统一版本管理 | 初期拆分工作量大 |
| C: 微服务化 | 独立部署、独立扩容 | 复杂度过高、团队规模不支撑 |

**决策**
选择 **方案 B：Maven 多模块聚合**。原因：
1. 团队规模小（1-3 人），微服务运维成本过高
2. 业务域之间耦合度中等，聚合足以解决问题
3. 通过 `mall-shared` 消灭重复代码，通过 `mall-modules` 保持领域边界

**影响**
- 所有公共能力必须放入 `mall-shared/*`
- 领域逻辑必须放入 `mall-modules/*`，BFF 保持薄
- 构建产出为 3 个独立可执行 JAR（app-api、admin-api、job）

### ADR-002: 前端采用 pnpm Monorepo

- **日期**：2026-02-23
- **状态**：✅ 已采纳
- **决策者**：架构团队

**背景**
V1/V2 的 admin-web 和 app-web 分别独立仓库，API 调用代码各写各的，导致：接口签名不一致、类型定义重复、版本依赖混乱。

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| A: 保持独立仓库 | 部署独立 | API 调用不一致、类型重复 |
| B: pnpm monorepo | 共享 API SDK、统一依赖、一处改全局生效 | 构建配置略复杂 |
| C: npm workspaces | 原理同 B | pnpm 性能更优、锁文件更可靠 |

**决策**
选择 **方案 B：pnpm monorepo**。`frontend/packages/api-sdk` 作为统一 API 封装层，所有前端请求必须通过它。

**影响**
- 禁止在页面层直接写 `axios` 调用
- 接口变更时先改 SDK，再改页面
- E2E 测试放在同一 workspace 中

### ADR-003: 无状态 JWT + Redis 黑名单认证

- **日期**：2026-02-23
- **状态**：✅ 已采纳
- **决策者**：架构团队

**背景**
需要选择前后端分离场景下的认证方案。要求：支持多端登录、无状态、支持即时登出。

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| A: Session + Cookie | 实现简单 | 有状态、跨域复杂、不支持 Native |
| B: JWT only | 无状态、支持多端 | 无法即时失效（除非等过期） |
| C: JWT + Redis 黑名单 | 无状态 + 即时登出 | 依赖 Redis |

**决策**
选择 **方案 C：JWT + Redis 黑名单**。`TokenBlacklistService` 在 logout 时将 token 写入 Redis（TTL=剩余有效期），`JwtAuthFilter` 在每次请求时检查黑名单。

**影响**
- Redis 成为认证链路的关键依赖
- 签名算法固定为 HS256
- 密钥通过 `${MALL_JWT_SECRET}` 环境变量注入

### ADR-004: Admin 端采用动态 RBAC 权限控制

- **日期**：2026-02-23
- **状态**：✅ 已采纳
- **决策者**：架构团队

**背景**
管理后台需要细粒度的权限控制（不同角色看到不同菜单和功能），且权限规则需要运行时可调整。

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| A: 硬编码角色判断 | 简单直接 | 每次改权限需要改代码重新部署 |
| B: Spring Security 注解 (@PreAuthorize) | 声明式 | 扩展性不够、规则分散在各 Controller |
| C: 数据库驱动动态 RBAC | 运行时可调整、集中管理 | 实现稍复杂、每次请求查 DB |

**决策**
选择 **方案 C：数据库驱动动态 RBAC**。`DynamicResourcePermissionService` 基于 `ums_resource` 表的 Ant Pattern URL 做运行时匹配。超级管理员 `admin` 账号直接放行。

**影响**
- 新增后台接口若需要权限管控，必须在 `ums_resource` 表中配置 URL Pattern
- App 端不使用此机制（使用简单的 `authenticated()`）

### ADR-005: ES 搜索降级 MySQL 兜底策略

- **日期**：2026-02-23
- **状态**：✅ 已采纳
- **决策者**：架构团队

**背景**
商品搜索需要全文检索能力（Elasticsearch），但 ES 可能因维护、资源不足等原因不可用。如果搜索完全依赖 ES，用户将无法搜索商品。

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| A: 仅 ES | 搜索体验最佳 | ES 不可用时搜索完全瘫痪 |
| B: 仅 MySQL LIKE | 无外部依赖 | 搜索质量差、大数据量性能差 |
| C: ES 优先 + MySQL 兜底 | 高可用、体验退化但不中断 | 需维护两套查询逻辑 |

**决策**
选择 **方案 C：ES 优先 + MySQL 兜底**。`SearchController` 在 ES 查询失败、返回空结果或命中未上架商品时，自动回退到 MySQL 条件查询。

**影响**
- `module-search` 不直接依赖 `module-product`，ES→MySQL 映射在 BFF 层完成
- 搜索结果排序质量在 ES 和 MySQL 模式下可能不同

### ADR-006: 支付模块 Mock 策略

- **日期**：2026-02-23
- **状态**：✅ 已采纳
- **决策者**：架构团队

**背景**
V3 初期无真实支付网关对接需求，但交易链路（下单→支付→状态流转）需要完整可调试。

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| A: 不实现支付 | 省开发量 | 交易链路断裂、前端无法联调 |
| B: 对接沙箱支付 | 最真实 | 依赖第三方、配置复杂、联调慢 |
| C: Mock 支付 | 无外部依赖、即时响应 | 不代表真实支付语义 |

**决策**
选择 **方案 C：Mock 支付**。`MockPaymentServiceImpl` 在创建支付单后立即标记支付成功并写入支付日志。当前实现不会在 `PaymentService` 内自动推进订单状态，订单状态流转需由订单接口显式触发（如 `/order/paySuccess`）。后续对接真实网关时，只需实现 `PaymentService` 接口的新实现类。

**影响**
- 支付相关 API 的 code=200 不代表真实扣款
- 生产上线前必须替换为真实支付实现

### ADR-007: 采用 Flyway 管理数据库版本

- **日期**：2026-02-23
- **状态**：✅ 已采纳
- **决策者**：架构团队

**背景**
V1/V2 使用手工 SQL 脚本管理数据库变更，无版本追踪、无冲突检测、团队协作易出错。

**方案对比**

| 方案 | 优势 | 劣势 |
|---|---|---|
| A: 手工 SQL | 灵活 | 无追踪、易冲突、回滚困难 |
| B: Flyway | 有版本管理、自动执行、冲突检测 | 不支持自动回滚 |
| C: Liquibase | 功能丰富、支持回滚 | XML 配置复杂 |

**决策**
选择 **方案 B：Flyway**。迁移脚本放在 `data/migration/`，种子数据放在 `data/seed/`，并保持 Flyway 命名规范；当前默认通过 `init-db.ps1` 脚本统一执行。

**实现现状（与代码对齐）**
- `mall-app-api` 与 `mall-admin-api` 当前配置为 `spring.flyway.enabled=false`
- 初始化流程由 `scripts/init-db.ps1` 按文件名顺序执行 `data/migration/*.sql` 与 `data/seed/*.sql`

**影响**
- 所有数据库变更必须通过新增迁移文件完成
- 回滚需手工编写反向 SQL
- 命名规范：`V{version}__{description}.sql`

## 4. 维护规则

1. 每条 ADR 的编号全局唯一、只增不减
2. 已采纳的 ADR 不修改原文；如果决策被推翻，新增 ADR 并将原 ADR 标记为 `❌ 已废弃` 或 `🔄 被 ADR-XXX 取代`
3. 做出重要架构决策时（技术选型、设计模式、中间件选择、数据模型变更），必须记录 ADR
4. 日常 Bug 修复、小优化不需要 ADR
